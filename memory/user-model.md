## Working style

- no destructive or irreversible action until it is verified end-to-end AND explicitly approved; prefer non-destructive paths (copy-not-move, new branch/worktree, additive/reversible edits) and minimize blast radius (evidence: 1, last: 2026-06-14, source: stated)
- when adapting another repo's artifact, COPY it into the target repo and edit the copy — never modify the source repo (evidence: 1, last: 2026-06-14, source: stated)
- iterate harness/skill changes in an isolated git worktree and test end-to-end before merging or replacing anything (evidence: 1, last: 2026-06-14, source: stated)
- when a subagent's or research's identity claim about the user's own thing (product name, owner, scope) conflicts with how the user framed the task, the user's framing wins — surface the conflict and reconcile, never silently re-label; a subagent reports what's written where you point it, not the user's intent. 2nd instance: an experiment was labeled "hangar" (the name in the stale competitor-scan docs) when the user had twice called the product "Grove" (evidence: 2, last: 2026-06-19, source: corrections)
- stay strictly inside the worktree/repo the user is working in: never read, search, or run in a SIBLING worktree (`.claude/worktrees/*`) or another repo — parallel chats/worktrees are deliberately isolated and cross-reaching corrupts their in-flight work (now mechanically enforced by hooks/guard_worktree_isolation.py + guard_worktree_session.py). Take stated scope LITERALLY — don't reinterpret or expand it ("stop trying to push it everywhere"); and never assert a fact, especially "X doesn't exist", from a partial/single-repo view — verify the real topology first, incl. config/topology facts ("X is already done"): a generated snapshot (e.g. the silo settings.json) does NOT auto-reflect later merges, so a prior init run is not proof the current state matches. Likewise a doc's status/header line ("Phase 0 built, Phases 1-4 proposed") is NOT proof of current build state — trust the code/artifact you actually read over a doc's self-description, especially in tools built to catch doc drift. A DERIVED/computed metric — a subagent's or your own freshly-written code's output ("N-node loop", "2 communities", a degree count) — is a CLAIM, not a fact: re-derive it (or show its inputs are deterministic) before stating it to the user, and label it by the assumptions it rests on — a "loop" built from documentation cross-references is NOT a control-flow loop (2026-06-19: presented an enrichment's "9-node self-improvement-loop SCC" as machine-truth; adversarial review showed it was a cites/nudges citation web with zero control edges) (evidence: 4, last: 2026-06-19, source: corrections)
- when the user has stated a clear disposition, act on it minimally — don't re-open it with stacked AskUserQuestion menus; repeated option-menus on a settled point read as runaround, not diligence (evidence: 1, last: 2026-06-17, source: corrections)
- never reference the private harness/brain in a PUBLIC or product repo's committed files (CLAUDE.md, README, .claude/) — harness coupling (route-to-trunk, predict-then-act cadence, retro targets) lives only in uncommitted global/local config; a committed CLAUDE.md in such a repo is pure product docs with zero harness mentions (evidence: 1, last: 2026-06-17, source: corrections)
- a terse git instruction (e.g. one-word "Branch") means take it to its natural, VISIBLE completion within the stated target (depth of the verb, NOT license to expand where the work goes) — for branch work that is push + open the PR, not an invented local-only "committed but not pushed" stop presented as a deliberate feature; this user's mental model of "make a branch" includes it being visible on the remote (evidence: 1, last: 2026-06-19, source: corrections)
